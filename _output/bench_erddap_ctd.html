Summary

CalCOFI’s CTD profile data is large in long (tidy) format — one row per cast × depth × measurement_type (~53 sensor types): ctd_measurement is ~216 M rows / ~16 GB of Parquet (partitioned by cruise_key), plus an adaptively-thinned headline table ctd_thin (~5.5 M rows). We wanted to serve these on ERDDAP — testing the native NetCDF path as well as backends that avoid converting away from Parquet — without the memory blow-up that EDDTableFromParquetFiles hits on large files.

We benchmarked four serving backends at a deliberately small 2 GB JVM heap:

backend EDDType format loads ctd_measurement (216 M)?
NetCDF (per-cruise CF profiles) EDDTableFromNcCFFiles wide ✅ (heap ~1.3 GB)
DuckDB (JDBC views over Parquet) EDDTableFromDatabase long ✅ (heap ~60 MB)
Parquet (one denormalized file) EDDTableFromParquetFiles long ❌ OOM / timeout
CSV (uncompressed) EDDTableFromAsciiFiles long ❌ (disk-prohibitive)

Headline findings:

  1. NetCDF is the native ERDDAP path and it works well — if the data is re-shaped to per-cruise CF profile files. CalCOFI has ~5.55 M casts, so “one NetCDF per cast” (à la Trinidad Head) is 5.55 M files (infeasible); bundling by cruise gives just 96 files. ERDDAP reads each file’s index and opens only matching files, so the 216 M-row-equivalent table loads in a ~1.3 GB heap and cruise-scoped queries are the fastest of any backend (it opens one file). ERDDAP also auto-computes actual_range from the files, so the min/max + sliders work with no extra config.
  2. DuckDB-over-Parquet keeps the tidy long table queryable as-is with a tiny footprint: a 274 KB .db of CREATE VIEW … read_parquet(…) views serves the full 216 M-row table within a ~60 MB load heap and streams filtered queries. It is live now at https://erddap.calcofi.io (calcofi_ctd_thin, calcofi_ctd_measurement).
  3. Plain Parquet is the memory trap. EDDTableFromParquetFiles reads each matching file fully into the JVM heap before filtering: a single 935 MB ctd_wide.parquet OOM’d a 4 GB heap, and the 216 M-row table won’t load at 2 GB.
  4. On the partitioning question (Lynn): for the typical “one variable in a lat/lon/depth box” query, partitioning the long table by measurement_type and ordering rows spatially within gives ~12× — quantified below — though every layout is already sub-second with DuckDB on SSD.

This note documents the benchmark, the layout experiment, and everything needed to reproduce it (§8) so others weighing Parquet/NetCDF-on-ERDDAP (cf. Roy Mendelssohn’s posts, ERDDAP #447 / #448) have concrete numbers.

1 · Background

CalCOFI has occupied the California Current quarterly since 1949. The CTD cast files (https://calcofi.org/data/oceanographic-data/ctd-cast-files/) are ~1 GB as a single wide Parquet (one column per sensor). We ingest them into a tidy long schema (ingest_calcofi_ctd-cast, https://calcofi.io/workflows/ingest_calcofi_ctd-cast.html): each sensor reading on its own row keyed by measurement_type, which is convenient for analysis and for adding new sensors, but multiplies row count. Tables (release v2026.06.08 / supplemental v2026.04.08):

table rows casts Parquet note
ctd_cast 5.55 M 5.55 M 128 MB one row per cast; carries time,lat,lon
ctd_thin 5.55 M 0.42 M 155 MB adaptively-thinned headline table
ctd_measurement 216 M 5.36 M ~16 GB full long table (96 cruise_key partitions)

The long tables are keyed by ctd_cast_uuid and do not carry coordinates — time/latitude/longitude live in ctd_cast. Any ERDDAP representation must re-join them. The 5.55 M casts number is what makes per-cast NetCDF infeasible and motivates the per-cruise bundling in §4.

2 · The memory problem with EDDTableFromParquetFiles

EDDTableFromParquetFiles (ERDDAP ≥ 2.27) reads each matching file fully into the JVM heap, then filters. That is fine for small files but:

  • The single 935 MB ctd_wide.parquet OOM’d our 4 GB heap and dropped the dataset from every LoadDatasets run.
  • Even partitioned and at a deliberately small 2 GB heap, the 216 M-row table could not load (heap peaked ~1.98 GB, then timed out building the file table / reading files).
  • Every query re-reads the whole matching file into heap, so latency scales with file size, not result size.

3 · DuckDB-over-Parquet (EDDTableFromDatabase)

ERDDAP has no native DuckDB type yet (EDDTableFromDuckDb, #448, is a proposal; the ADBC/S3 idea is #447). But ERDDAP already speaks JDBC via EDDTableFromDatabase, and DuckDB ships a JDBC driver. So we:

  1. Build a tiny .db (274 KB — view definitions only) of CREATE VIEW ctd_*_erddap AS SELECT … FROM read_parquet('…/ctd_*/**/*.parquet') t JOIN read_parquet('…/ctd_cast.parquet') c USING (ctd_cast_uuid) — the join denormalizes time/lat/lon logically (no physical 16 GB rewrite).
  2. Add the DuckDB JDBC jar to ERDDAP’s WEB-INF/lib via a 4-line custom image (FROM erddap/erddap:latest). Pin the jar to the DuckDB engine that built the .db (we use 1.5.2).
  3. Point EDDTableFromDatabase at jdbc:duckdb:/…/calcofi_ctd.db with connection properties duckdb.read_only=true, memory_limit, threads, temp_directory, and sourceCanOrderBy/sourceCanDoDistinct = yes.

DuckDB then does predicate pushdown + Parquet partition pruning + row-group skipping, and streams the JDBC ResultSet back to ERDDAP — so the whole table is never on the heap.

ImportantGotcha: serve time as a TIMESTAMP, not epoch double

ERDDAP reads any time variable (units seconds since 1970-…) via ResultSet.getTimestamp(). If the DuckDB view exposes time as an epoch DOUBLE, the DuckDB JDBC driver throws NullPointerException: … varlen_data is null on every query that selects time. Expose time as a real TIMESTAMP in the view (ERDDAP converts it to epoch seconds) and run the container in UTC. Also: EDDTableFromDatabase does not auto-compute variable ranges, so set actual_range (+ geospatial_*/time_coverage_*) on time/lat/lon/depth or the form shows no min/max or sliders. (NetCDF, §4, computes these for you.)

4 · NetCDF (EDDTableFromNcCFFiles) — the native ERDDAP path

This is Lynn’s first option (“NetCDF per cast, à la your Trinidad Head setup”) and ERDDAP’s bread-and-butter format. The catch is scale: CalCOFI has ~5.55 M casts, so one file per cast is 5.55 M files — far past what ERDDAP’s file table can manage. We bundle by cruise instead: 96 files, one per cruise_key.

Each file is a CF Discrete-Sampling-Geometry “profile” dataset (contiguous ragged array): profile = cast, obs = depth level, canonical sensors pivoted wide to one variable each. We generate them by pivoting each cruise’s long rows in DuckDB and writing with R ncdf4 (libs/erddap_netcdf.R, scripts/gen_ctd_netcdf.R), one cruise at a time so memory stays bounded.

Two things fall out nicely:

  • ERDDAP auto-computes actual_range for time/lat/lon/depth from the files — so the min/max + sliders work with no manual ranges (unlike DuckDB above).
  • accessibleViaFiles=true lets users download the source .nc directly.

Trade-offs to weigh: the wide files carry canonical sensors only (non-canonical variants are dropped); they duplicate the data (~2.3 GB of NetCDF for the full table); and they must be regenerated when the data updates (vs the DuckDB view, which always reflects the live Parquet). CalCOFI’s casts are also fine-grained (≈1 depth per ctd_cast_uuid), so these are effectively point-like profiles.

Takeaways. Both NetCDF and DuckDB load both tables (incl. 216 M rows) within the 2 GB cap; Parquet cannot load the big table and CSV cannot even load the thin one. DuckDB’s load heap is flattest (~60 MB) because it streams from the live Parquet; NetCDF’s is higher (~1.3 GB) because ERDDAP reads all 95 file headers, but it still fits — and NetCDF then serves cruise-scoped queries fastest of all (q4 dump on the big table: 0.8 s vs DuckDB’s 99 s), since one cruise_key maps to one file. The catch is that the wide/canonical NetCDF returns less data per dump than the long backends (which emit every sensor row), so q4 is not a like-for-like byte comparison — q2/q3, where result sizes match closely, are the fair head-to-head, and there NetCDF and DuckDB trade the lead.

6 · Partitioning & ordering for long-format bbox queries

“I wonder how organizing each sensor reading on its own row affects a query for data within a lat/lon/depth box.” — Lynn DeWitt

This is the key question for long format. We measured (DuckDB, the serving engine) the same bounding-box query — temperature_ave, 32–34°N, 119–121°W, 0–200 m (~393 k rows) — against four physical layouts of the 216 M-row denormalized table:

  • A — production view: raw Parquet partitioned by cruise_key, joined to ctd_cast at query time (what’s live now).
  • B — denormalized, partitioned by cruise_key (isolates layout from the join).
  • C — denormalized, partitioned by measurement_type (one variable ⇒ ~1/N files).
  • D — denormalized, partitioned by measurement_type and ordered by latitude, longitude, depth within each partition (row-group skipping for the box).

How to read this. Partition pruning (C, D) drops the data DuckDB must open from all ~95 cruise files to just the one measurement_type partition. Spatial ordering (D) additionally lets DuckDB skip Parquet row groups whose lat/lon/depth min–max can’t intersect the box. Decomposing the ~12× spread:

  • Removing the query-time join ≈ 4× (A→B): materializing time/lat/lon into the table skips the join and lets the lat/lon predicate prune at scan.
  • Partitioning by measurement_type ≈ 2× (B→C): a one-variable query opens 1 of 53 partitions instead of all 95 cruise files.
  • Spatial ordering within each partition ≈ 1.4× (C→D): DuckDB skips Parquet row groups whose latitude/longitude/depth min–max miss the box.

The trade-off is exactly your point: partitioning by measurement_type helps single-variable queries (the common case) but slows a query that wants all variables in a region. Our live datasets keep the zero-extra-storage logical-join view (layout A, 0.2–0.9 s here); if heavy single-variable bbox traffic emerges, the measurement_type-partition + spatial-sort layout is a cheap ~12× upgrade. (This is the same wide-vs-access-pattern tension that makes the per-cruise NetCDF wide files fast for “all sensors at a cast” but slower for “one sensor across a region.”)

7 · Recommendations & open questions

  • If you host on CoastWatch ERDDAP, NetCDF is the most native fit — EDDTableFromNcCFFiles over per-cruise CF profile files (96 files), which we generate from the same pipeline. It loads within a small heap, auto-computes ranges/sliders, and serves cruise-scoped queries fastest. Costs: it carries canonical sensors only, duplicates ~2.3 GB, and must be regenerated on update.
  • If you want the tidy long table queryable as-is, DuckDB-over-Parquet is the lightest option: a 274 KB .db of views on stock ERDDAP (only the driver jar is added), serving the 216 M-row table within ~60 MB. It always reflects the live Parquet. It’s what runs at https://erddap.calcofi.io today.
  • Avoid plain EDDTableFromParquetFiles for large files — whole-file on-heap reads OOM. Both options above are pragmatic bridges to native Parquet/DuckDB support (#447/#448).
  • Layout: for the full long table, partition by measurement_type and order spatially within if single-variable bbox access dominates.
  • Open for you, Lynn: which would be least work to receive and maintain —
    1. per-cruise NetCDF files we generate, (b) Parquet + the small DuckDB .db (this approach), or (c) waiting on native DuckDB/ADBC? We can produce any from the same pipeline.

# erddap_netcdf.R — publish CalCOFI long CTD tables to ERDDAP as the NATIVE
# ERDDAP path: per-cruise CF "profile" NetCDF files served by EDDTableFromNcCFFiles
# (Discrete Sampling Geometry, contiguous ragged array). This is the "NetCDF per
# cast à la Trinidad Head" option Lynn DeWitt raised — except CalCOFI has ~5.55M
# casts, so one file PER CAST is infeasible; we bundle casts BY CRUISE (96 files).
#
# The long tables (one row per cast x depth x measurement_type) are pivoted to
# WIDE (one column per canonical measurement_type) per cruise, joined to ctd_cast
# for time/lat/lon, and written as CF profiles: profile = cast, obs = depth level.
# ERDDAP reads each file's per-file index and only opens matching files, so memory
# stays low (contrast EDDTableFromParquetFiles, which loads whole files on-heap).
#
# Reuses erddap.R helpers (.erddap_datavar_xml, .xml_escape, %||%).
# Requires: DBI, duckdb, glue, ncdf4. Generate in the rstudio container, reading
# Parquet from and writing NetCDF to the shared /share/data tree (so the paths
# resolve identically when ERDDAP later reads them). See bench_erddap_ctd.qmd.

# CF/NetCDF default double fill (recognized by ERDDAP as the missing value).
.NC_FILL_DOUBLE <- 9.969209968386869e36

#' Pivot one cruise of a long CTD table to a wide (cast x depth) data.frame.
#' @return data.frame ordered by (profile_id, depth): columns profile_id, time
#'   (epoch s), latitude, longitude, line, sta, depth, then one per `vars`.
.ctd_cruise_wide <- function(con, data_dir, table, cruise_key, vars) {
  val <- vapply(vars, function(v) glue::glue(
    "MAX(t.measurement_value) FILTER (WHERE t.measurement_type='{v}')::DOUBLE AS \"{v}\""),
    character(1))
  sql <- glue::glue(
    "SELECT
       t.ctd_cast_uuid                                  AS profile_id,
       epoch(any_value(c.datetime_start_utc))::DOUBLE   AS time,
       any_value(c.latitude)::DOUBLE                    AS latitude,
       any_value(c.longitude)::DOUBLE                   AS longitude,
       any_value(c.line)                                AS line,
       any_value(c.sta)                                 AS sta,
       t.depth_m::DOUBLE                                AS depth,
       {paste(val, collapse=',\n       ')}
     FROM read_parquet('{data_dir}/{table}/**/*.parquet', hive_partitioning=true) t
     JOIN read_parquet('{data_dir}/ctd_cast.parquet') c USING (ctd_cast_uuid)
     WHERE t.cruise_key = '{cruise_key}'
     GROUP BY t.ctd_cast_uuid, t.depth_m
     ORDER BY t.ctd_cast_uuid, t.depth_m")
  DBI::dbGetQuery(con, sql)
}

#' Write one cruise's wide data.frame to a CF contiguous-ragged Profile NetCDF.
.write_cruise_nc <- function(df, cruise_key, out_file, vars, title, summary,
                             units_lookup = list(), longname_lookup = list(),
                             institution = "CalCOFI") {
  prof_ids  <- unique(df$profile_id)          # df is ordered by profile_id, so contiguous
  nprof     <- length(prof_ids)
  nobs      <- nrow(df)
  rowSize   <- as.integer(table(factor(df$profile_id, levels = prof_ids)))
  first_idx <- match(prof_ids, df$profile_id) # first obs row of each profile
  STRLEN    <- 64L
  FILL      <- .NC_FILL_DOUBLE

  d_obs  <- ncdf4::ncdim_def("obs",     "", seq_len(nobs),  create_dimvar = FALSE)
  d_prof <- ncdf4::ncdim_def("profile", "", seq_len(nprof), create_dimvar = FALSE)
  d_str  <- ncdf4::ncdim_def("name_strlen", "", seq_len(STRLEN), create_dimvar = FALSE)

  # profile-level (indexed by profile)
  v_pid   <- ncdf4::ncvar_def("profile_id", "", list(d_str, d_prof), prec = "char")
  v_cru   <- ncdf4::ncvar_def("cruise_key", "", list(d_str, d_prof), prec = "char")
  v_line  <- ncdf4::ncvar_def("line",       "", list(d_str, d_prof), prec = "char")
  v_sta   <- ncdf4::ncvar_def("sta",        "", list(d_str, d_prof), prec = "char")
  v_time  <- ncdf4::ncvar_def("time", "seconds since 1970-01-01T00:00:00Z", d_prof, prec = "double")
  v_lat   <- ncdf4::ncvar_def("latitude",  "degrees_north", d_prof, prec = "double")
  v_lon   <- ncdf4::ncvar_def("longitude", "degrees_east",  d_prof, prec = "double")
  v_rs    <- ncdf4::ncvar_def("rowSize",   "",              d_prof, prec = "integer")
  # obs-level (indexed by obs)

#' Build per-cruise CF Profile NetCDF for a long CTD table.
#' @param data_dir base dir with {table}/ (hive by cruise_key) + ctd_cast.parquet.
#' @param out_dir  destination for <cruise_key>.nc (created if absent).
#' @param vars     canonical measurement_type names -> wide NetCDF variables.
#' @param cruises  optional subset of cruise_keys (default: all partitions).
#' @param con      a live DuckDB DBI connection (caller-managed; reused so the
#'   :memory: instance lifecycle stays in one place).
#' @param overwrite re-write existing .nc (default FALSE -> resumable).
#' @return data.frame(cruise_key, nprof, nobs, file, mb) for files written/seen.
build_ctd_netcdf <- function(con, data_dir, out_dir, table, vars, title, summary,
                             units_lookup = list(), longname_lookup = list(),
                             cruises = NULL, mem_limit = "2GB", threads = 2,
                             tmp_dir = "/share/data/erddap-duckdb/tmp",
                             overwrite = FALSE, verbose = TRUE) {
  if (!dir.exists(out_dir)) dir.create(out_dir, recursive = TRUE)
  if (is.null(cruises)) {
    parts <- list.files(file.path(data_dir, table), pattern = "^cruise_key=")
    cruises <- sort(sub("^cruise_key=", "", parts))
  }
  cruises <- setdiff(cruises, "__HIVE_DEFAULT_PARTITION__")  # NULL-cruise hive bucket
  for (s in c(glue::glue("SET memory_limit='{mem_limit}'"),
              glue::glue("SET threads={threads}"),
              glue::glue("SET temp_directory='{tmp_dir}'"),
              "SET preserve_insertion_order=false"))
    try(DBI::dbExecute(con, s), silent = TRUE)
  rows <- vector("list", length(cruises))
  for (i in seq_along(cruises)) {
    ck   <- cruises[i]
    f    <- file.path(out_dir, paste0(ck, ".nc"))
    if (file.exists(f) && !overwrite) {
      rows[[i]] <- data.frame(cruise_key = ck, nprof = NA, nobs = NA,
                              file = f, mb = round(file.size(f) / 1048576, 2))
      if (verbose) cat(sprintf("[%d/%d] %s  (exists, skip)\n", i, length(cruises), ck))
      next
    }
    df  <- .ctd_cruise_wide(con, data_dir, table, ck, vars)
    if (nrow(df) == 0L) {                       # no casts join to ctd_cast — skip
      if (verbose) cat(sprintf("[%d/%d] %s  (0 rows, skip)\n", i, length(cruises), ck))
      next
    }
    dims <- .write_cruise_nc(df, ck, f, vars, title, summary, units_lookup, longname_lookup)
    rows[[i]] <- data.frame(cruise_key = ck, nprof = dims["nprof"], nobs = dims["nobs"],
                            file = f, mb = round(file.size(f) / 1048576, 2))
    if (verbose) cat(sprintf("[%d/%d] %s  profiles=%d obs=%d  %.1f MB\n",
                             i, length(cruises), ck, dims["nprof"], dims["nobs"],
                             file.size(f) / 1048576))
    rm(df); gc(FALSE)
  }
  do.call(rbind, rows)
}

## Summary

CalCOFI's CTD profile data is large in *long* (tidy) format — one row per
cast × depth × `measurement_type` (~53 sensor types): `ctd_measurement` is
**~216 M rows / ~16 GB** of Parquet (partitioned by `cruise_key`), plus an
adaptively-thinned headline table `ctd_thin` (~5.5 M rows). We wanted to serve
these on ERDDAP — testing the **native NetCDF path** as well as backends that
avoid converting away from Parquet — without the memory blow-up that
`EDDTableFromParquetFiles` hits on large files.

We benchmarked four serving backends at a deliberately small **2 GB JVM heap**:

| backend | EDDType | format | loads `ctd_measurement` (216 M)? |
|---|---|---|:--:|
| **NetCDF** (per-cruise CF profiles) | `EDDTableFromNcCFFiles` | wide | ✅ (heap ~1.3 GB) |
| **DuckDB** (JDBC views over Parquet) | `EDDTableFromDatabase` | long | ✅ (heap ~60 MB) |
| **Parquet** (one denormalized file) | `EDDTableFromParquetFiles` | long | ❌ OOM / timeout |
| **CSV** (uncompressed) | `EDDTableFromAsciiFiles` | long | ❌ (disk-prohibitive) |

**Headline findings:**

1. **NetCDF is the native ERDDAP path and it works well** — *if* the data is
   re-shaped to per-cruise CF profile files. CalCOFI has **~5.55 M casts**, so
   "one NetCDF *per cast*" (à la Trinidad Head) is **5.55 M files** (infeasible);
   bundling **by cruise gives just 96 files**. ERDDAP reads each file's index and
   opens only matching files, so the 216 M-row-equivalent table loads in a ~1.3 GB
   heap and **cruise-scoped queries are the fastest of any backend** (it opens one
   file). ERDDAP also auto-computes `actual_range` from the files, so the
   min/max + sliders work with no extra config.
2. **DuckDB-over-Parquet keeps the tidy long table queryable as-is** with a tiny
   footprint: a 274 KB `.db` of `CREATE VIEW … read_parquet(…)` views serves the
   **full 216 M-row table within a ~60 MB load heap** and streams filtered queries.
   It is live now at <https://erddap.calcofi.io> (`calcofi_ctd_thin`,
   `calcofi_ctd_measurement`).
3. **Plain Parquet is the memory trap.** `EDDTableFromParquetFiles` reads each
   matching file *fully into the JVM heap* before filtering: a single 935 MB
   `ctd_wide.parquet` OOM'd a 4 GB heap, and the 216 M-row table won't load at 2 GB.
4. On the **partitioning** question (Lynn): for the typical *"one variable in a
   lat/lon/depth box"* query, partitioning the long table by `measurement_type`
   and ordering rows spatially within gives ~12× — quantified below — though every
   layout is already sub-second with DuckDB on SSD.

to reproduce it** (§8) so others weighing Parquet/NetCDF-on-ERDDAP (cf. Roy
Mendelssohn's posts, ERDDAP
[#447](https://github.com/ERDDAP/erddap/discussions/447) /
[#448](https://github.com/ERDDAP/erddap/issues/448)) have concrete numbers.

**long** schema (`ingest_calcofi_ctd-cast`,
<https://calcofi.io/workflows/ingest_calcofi_ctd-cast.html>): each sensor reading
on its own row keyed by `measurement_type`, which is convenient for analysis and
for adding new sensors, but multiplies row count. Tables (release v2026.06.08 /
supplemental v2026.04.08):

| table | rows | casts | Parquet | note |
|---|--:|--:|--:|---|
| `ctd_cast` | 5.55 M | 5.55 M | 128 MB | one row per cast; carries `time`,`lat`,`lon` |
| `ctd_thin` | 5.55 M | 0.42 M | 155 MB | adaptively-thinned headline table |
| `ctd_measurement` | 216 M | 5.36 M | ~16 GB | full long table (96 `cruise_key` partitions) |

The long tables are keyed by `ctd_cast_uuid` and **do not carry coordinates** —
`time`/`latitude`/`longitude` live in `ctd_cast`. Any ERDDAP representation must
re-join them. The **5.55 M casts** number is what makes per-cast NetCDF
infeasible and motivates the per-cruise bundling in §4.

## 2 · The memory problem with `EDDTableFromParquetFiles`

`EDDTableFromParquetFiles` (ERDDAP ≥ 2.27) reads each *matching file* fully into
the JVM heap, then filters. That is fine for small files but:

- The single **935 MB** `ctd_wide.parquet` OOM'd our 4 GB heap and dropped the
  dataset from every `LoadDatasets` run.
- Even partitioned and at a deliberately small 2 GB heap, the 216 M-row table
  **could not load** (heap peaked ~1.98 GB, then timed out building the file
  table / reading files).
- Every *query* re-reads the whole matching file into heap, so latency scales
  with file size, not result size.

## 3 · DuckDB-over-Parquet (`EDDTableFromDatabase`)

ERDDAP has no native DuckDB type yet (`EDDTableFromDuckDb`,
[#448](https://github.com/ERDDAP/erddap/issues/448), is a proposal; the ADBC/S3
idea is [#447](https://github.com/ERDDAP/erddap/discussions/447)). But ERDDAP
*already* speaks JDBC via `EDDTableFromDatabase`, and DuckDB ships a JDBC driver.
So we:

1. Build a **tiny `.db`** (274 KB — view definitions only) of
   `CREATE VIEW ctd_*_erddap AS SELECT … FROM read_parquet('…/ctd_*/**/*.parquet')
   t JOIN read_parquet('…/ctd_cast.parquet') c USING (ctd_cast_uuid)` — the join
   denormalizes `time`/`lat`/`lon` *logically* (no physical 16 GB rewrite).
2. Add the **DuckDB JDBC jar** to ERDDAP's `WEB-INF/lib` via a 4-line custom
   image (`FROM erddap/erddap:latest`). Pin the jar to the DuckDB engine that
   built the `.db` (we use 1.5.2).
3. Point `EDDTableFromDatabase` at `jdbc:duckdb:/…/calcofi_ctd.db` with
   connection properties `duckdb.read_only=true, memory_limit, threads,
   temp_directory`, and `sourceCanOrderBy/sourceCanDoDistinct = yes`.

DuckDB then does predicate pushdown + Parquet partition pruning + row-group
skipping, and streams the JDBC `ResultSet` back to ERDDAP — so the whole table is
never on the heap.

::: {.callout-important title="Gotcha: serve time as a TIMESTAMP, not epoch double"}
ERDDAP reads any `time` variable (units `seconds since 1970-…`) via
`ResultSet.getTimestamp()`. If the DuckDB view exposes `time` as an epoch
**DOUBLE**, the DuckDB JDBC driver throws `NullPointerException: … varlen_data is
null` on *every* query that selects time. Expose `time` as a real **TIMESTAMP**
in the view (ERDDAP converts it to epoch seconds) and run the container in UTC.
Also: `EDDTableFromDatabase` does **not** auto-compute variable ranges, so set
`actual_range` (+ `geospatial_*`/`time_coverage_*`) on `time`/`lat`/`lon`/`depth`
or the form shows no min/max or sliders. (NetCDF, §4, computes these for you.)
:::

## 4 · NetCDF (`EDDTableFromNcCFFiles`) — the native ERDDAP path

This is Lynn's first option ("NetCDF per cast, à la your Trinidad Head setup") and
ERDDAP's bread-and-butter format. The catch is **scale**: CalCOFI has **~5.55 M
casts**, so one file per cast is 5.55 M files — far past what ERDDAP's file table
can manage. We bundle **by cruise** instead: **96 files**, one per `cruise_key`.

Each file is a **CF Discrete-Sampling-Geometry "profile"** dataset (contiguous
ragged array): `profile = cast`, `obs = depth level`, canonical sensors pivoted
**wide** to one variable each. We generate them by pivoting each cruise's long
rows in DuckDB and writing with R `ncdf4` (`libs/erddap_netcdf.R`,
`scripts/gen_ctd_netcdf.R`), one cruise at a time so memory stays bounded.

Two things fall out nicely:

- **ERDDAP auto-computes `actual_range`** for time/lat/lon/depth from the files —
  so the min/max + sliders work with **no** manual ranges (unlike DuckDB above).
- **`accessibleViaFiles=true`** lets users download the source `.nc` directly.

Trade-offs to weigh: the wide files carry **canonical sensors only** (non-canonical
variants are dropped); they **duplicate** the data (~2.3 GB of NetCDF for the full
table); and they must be **regenerated** when the data updates (vs the DuckDB view,
which always reflects the live Parquet). CalCOFI's casts are also fine-grained
(≈1 depth per `ctd_cast_uuid`), so these are effectively point-like profiles.

## 6 · Partitioning & ordering for long-format bbox queries

> *"I wonder how organizing each sensor reading on its own row affects a query for
> data within a lat/lon/depth box."* — Lynn DeWitt

This is the key question for long format. We measured (DuckDB, the serving engine)
the same bounding-box query — **`temperature_ave`, 32–34°N, 119–121°W, 0–200 m
(~393 k rows)** — against four physical layouts of the 216 M-row denormalized
table:

- **A** — production view: raw Parquet partitioned by `cruise_key`, joined to
  `ctd_cast` at query time (what's live now).
- **B** — denormalized, partitioned by `cruise_key` (isolates layout from the join).
- **C** — denormalized, partitioned by `measurement_type` (one variable ⇒ ~1/N files).
- **D** — denormalized, partitioned by `measurement_type` **and** ordered by
  `latitude, longitude, depth` within each partition (row-group skipping for the box).

**How to read this.** Partition pruning (C, D) drops the data DuckDB must open
from all ~95 cruise files to just the one `measurement_type` partition. Spatial
ordering (D) additionally lets DuckDB skip Parquet **row groups** whose
lat/lon/depth min–max can't intersect the box. Decomposing the ~12× spread:

- **Removing the query-time join ≈ 4×** (A→B): materializing `time`/`lat`/`lon`
  into the table skips the join *and* lets the lat/lon predicate prune at scan.
- **Partitioning by `measurement_type` ≈ 2×** (B→C): a one-variable query opens
  1 of 53 partitions instead of all 95 cruise files.
- **Spatial ordering within each partition ≈ 1.4×** (C→D): DuckDB skips Parquet
  row groups whose `latitude/longitude/depth` min–max miss the box.

The trade-off is exactly your point: partitioning by `measurement_type` *helps
single-variable queries* (the common case) but *slows* a query that wants *all*
variables in a region. Our live datasets keep the **zero-extra-storage
logical-join view** (layout A, 0.2–0.9 s here); if heavy single-variable bbox
traffic emerges, the `measurement_type`-partition + spatial-sort layout is a cheap
~12× upgrade. (This is the same wide-vs-access-pattern tension that makes the
per-cruise *NetCDF* wide files fast for "all sensors at a cast" but slower for "one
sensor across a region.")

## 7 · Recommendations & open questions

- **If you host on CoastWatch ERDDAP, NetCDF is the most native fit** —
  `EDDTableFromNcCFFiles` over **per-cruise CF profile files (96 files)**, which we
  generate from the same pipeline. It loads within a small heap, auto-computes
  ranges/sliders, and serves cruise-scoped queries fastest. Costs: it carries
  canonical sensors only, duplicates ~2.3 GB, and must be regenerated on update.
- **If you want the tidy long table queryable as-is, DuckDB-over-Parquet** is the
  lightest option: a 274 KB `.db` of views on stock ERDDAP (only the driver jar is
  added), serving the 216 M-row table within ~60 MB. It always reflects the live
  Parquet. It's what runs at <https://erddap.calcofi.io> today.
- **Avoid plain `EDDTableFromParquetFiles` for large files** — whole-file on-heap
  reads OOM. Both options above are pragmatic bridges to native Parquet/DuckDB
  support (#447/#448).
- **Layout:** for the full long table, partition by `measurement_type` and order
  spatially within if single-variable bbox access dominates.
- **Open for you, Lynn:** which would be least work to receive and maintain —
  (a) per-cruise **NetCDF** files we generate, (b) **Parquet + the small DuckDB
  `.db`** (this approach), or (c) waiting on native DuckDB/ADBC? We can produce any
  from the same pipeline.
